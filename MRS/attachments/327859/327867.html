Unit Testing Guidelines - TRIPS replacement

  • The developer who writes the class is responsible for making sure a corresponding test class gets written. Ideally, another developer should write the test class and try to 'break' the methods.
  • Each package to have a sub-package called tests which contains all the testing classes.
  • There is a class called TestAll which runs all the tests for the package.
  • Tests should be incorporated into the build process, so any modifications made are tested automatically. (put into ant build.xml script)

JUnit

  • Java Utility classes to make unit testing easy to implement.
  • Allows a test class to be written, which implements the TestCase class of Junit.
  • Methods to override: Override setUp() to initalize the test variables.
    Override setDown() to release any permanent resource you allocated in setUp()
  • test methods use start with "test" to be recognized by JUnit.
  • each test method should be a single test case.
  • each test method must end in an assert method being called. This simply tests your assertion in the test case and if fails, will provide the nessasary feedback to JUnit. There are a few basic types: assert(boolean)
    assertEquals()
    assertNull()
    assertNotNull()
    assertSame()
     the assertEquals method can handle all primitive types, check the api
  • more details info on Junit, read the primer
    This explains how a test suite can be made, which should be a packages TestAll class
  • Follow the template given, and check the api for more info.
  • Tests are run by two ways: java junit.textui.TestRunner <test-class>
    java junit.swingui.TestRunner <test-class>

EJB

  • Test individual entity beans first to ensure a functional data layer
  • Then test business logic tier of session beans
  • Get the home interface of the ejb in the setUp method.
  • Create a method which returns a new instance of the ejb for each test
  • See the example.
